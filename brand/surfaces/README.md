# Sora Surfaces

Surface files define how the Sora brand system appears across applied touchpoints. They translate the core brand strategy, product logic, voice, and visual direction into specific formats such as campaign panels, social posts, website sections, presentation slides, and product captions.

These files are designed to support the course assignments that test whether the Sora brand can scale across formats without losing coherence.

## Assignment Fit

### Assignment 6: Multi-Touchpoint Stress Test
Use these files to generate, compare, and evaluate two distinct touchpoints.

Recommended pair:

- **Static touchpoint:** `campaign-panel.md`
- **Dynamic touchpoint:** `social.md` or `website.md`, depending on whether the second execution is a motion/social loop or a digital interface/product explainer

Assignment 6 requires:

- Two distinct brand applications
- Adjusted inputs for each format
- A translation map explaining what stayed consistent and what changed
- A governance check that can evaluate brand consistency

The surface files provide the format-specific rules needed for those steps.

### Final Project: AI-Directed Brand Book
Use these files to build the Visual World, System Logic, Governance, and Appendix portions of the final brand book.

Recommended use:

- `headline.md` for slide titles, campaign language, and hero statements
- `campaign-panel.md` for high-impact static applications
- `social.md` for small-scale and motion-oriented applications
- `website.md` for UI/digital layout examples
- `presentation-slide.md` for the final deck itself
- `product-caption.md` for explanatory labels, system captions, and product logic descriptions

## Surface Principles

All Sora surfaces should feel:

- **Adaptive, not futuristic for its own sake**
- **Climate-aware, not fear-based**
- **Residential and practical, not speculative sci-fi**
- **Elegant but understandable**
- **System-driven, not random or decorative**

## Universal Requirements

Across every surface:

1. The product must remain a responsive roofing material.
2. The system must respond to **roof surface temperature**, not weather forecasts, time of day, indoor temperature, or user mood.
3. Reflect Mode and Absorb Mode must be visually distinct but belong to one material system.
4. Sora should be presented as practical climate adaptation, not luxury tech or generic sustainability branding.
5. Visuals should preserve architectural realism and avoid obvious AI artifacts.
6. Copy should be calm, direct, and grounded.

## Surface File Roles

| File | Role |
|---|---|
| `headline.md` | Defines campaign and presentation headline behavior. |
| `social.md` | Defines rules for small-scale, feed, story, and motion/social applications. |
| `website.md` | Defines landing page, product explainer, and UI-style surface rules. |
| `presentation-slide.md` | Defines how Sora appears in class presentations and the final brand book. |
| `campaign-panel.md` | Defines high-impact static panels, posters, billboards, and environmental layouts. |
| `product-caption.md` | Defines short explanatory copy for product diagrams, captions, callouts, and labels. |

## Governance Use

When evaluating any surface, ask:

- Does this still clearly communicate Sora as a responsive roofing material?
- Does the surface preserve the warm, practical, climate-adaptive tone?
- Are Reflect Mode and Absorb Mode accurate and visually controlled?
- Has the AI added generic environmental, smart-home, sci-fi, or decorative elements that weaken the brand?
- Would this surface belong in the same system as the hero product imagery?
